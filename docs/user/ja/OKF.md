# OKF — Open Knowledge Format

最終更新: 2026-07-07

OKF（Open Knowledge Format）は、Markdownの知識コレクションのためのオープンな規約です: 小さく統一されたFrontmatterヘッダーを持つ、純粋なMarkdownファイルです。このページでは、OKFとは何か、Plainvaが自動的に何を行うのか——そして、なぜあなたがそれらを*使う必要がない*のかを説明します。

## OKFとは？

その考え方は: 保管庫内のすべてのドキュメントが、それ自身で何であるかを示すということです。そのためには、Frontmatterの最小限のヘッダーだけで十分です。

```markdown
---
type: Note
okf_version: "0.1"
---
# My note
```

- **`type`**——このドキュメントがどのような種類か（例: `Note`、`Daily Note`、`Project`）。規約の唯一の必須フィールドです。
- **`okf_version`**——ファイルがどのバージョンの規約に基づいて書かれたかを示します。
- **`index.md`**——各フォルダーは、その目次として1つの`index.md`を持つことができます。`index.md`と`log.md`という名前はこのために予約されており、通常のノートには使用すべきではありません。

> ツールやスクリプトでファイルを書きますか？ 正確なフィールド契約——許可される値、各プロパティタイプがどのようにシリアライズされるか、予約名のルール——は[ファイル形式リファレンス](File_Format_Reference.md)にあります。

## なぜPlainvaはOKFを使うのか？

純粋なMarkdownは驚くほどポータブルです——しかしそれだけでは信頼できる構造を持ちません。OKFはそこに必要最小限の構造を加えるだけで、すべては標準的なFrontmatterを持つ、ごく普通のMarkdownのままです:

- **データベース、フィルター、テンプレートは構造に依拠できます。** すべてのノートが`type`を持つため、普通のファイルに対する`.base`ビューも堅牢であり続けます。
- **フォルダーはナビゲート可能なままです。** フォルダーごとの`index.md`という目次は、人にもツールにも役立ちます。
- **スクリプトやAIアシスタントも保管庫を安全に扱えます**——ディスク上の形式が統一され、文書化されているためです。
- **ロックインなし。** OKFは、ごく普通のMarkdownの上に成り立つオープンな規約です——他のOKFツールも、今日そして10年後もあなたのファイルを理解できます。

## Plainvaが自動的に行うこと

**新規ファイル**にはOKFヘッダーが自動的に付与されます: Plainvaで作成されたすべてのノートは、そのFrontmatterに`type`と`okf_version`を受け取ります。値は保管庫ごとに設定できます: **設定 → 保管庫の設定 → OKF（Open Knowledge Format）** → **新規ノートのtype**（デフォルトは`Note`）と**デイリーノートのtype**（デフォルトは`Daily Note`）。テンプレートが独自の`type`を持っている場合は、テンプレートが優先されます。

**既存のファイルは無断で変更されることはありません。** Plainvaは、新規ファイルを作成する際、またはあなたが明示的に変換を開始した場合にのみ、OKFフィールドを追加します。

**保護されたシステムフィールド:** **プロパティ**パネルでは、`type`と`okf_version`がOKFシステムフィールド（「OKFシステムフィールド——Plainvaによって管理されています」）としてマークされています: `type`の値は既知のタイプのドロップダウンから選択でき、`okf_version`は表示のみです。規約が誤って壊れないよう、名前変更、タイプの変更、削除はロックされています。

**説明モーダル:** 保管庫を初めて開くと、Plainvaは一度だけ**OKFとは？**を表示します——同じ要約は、いつでも設定内で確認できます。

## index.md: フォルダーごとの目次

`index.md`はフォルダーの目次です: そのフォルダーに含まれるノートとサブフォルダーの一覧で、説明と相対リンクが付いています。

- **生成**——常にあなたの操作に応じて、何もないところから自動的に生成されることはありません: フォルダーを右クリック → **index.mdを生成/更新**、またはまとめて**index.md管理**（**設定 → OKF → 開く…**）から行います。
- **生成の代わりに採用する**——既に概要ノート（MOC、概要、フォルダーノート、READMEなど）がある場合、管理画面がそれらを候補として提案します。**採用**は、そのファイルを`index.md`に名前変更し（リンクは保管庫全体で更新されます）、任意でOKF向けに整えることもできます。
- **自動的な保守**——Plainvaによって*生成された*一覧には、ファイルの末尾に不可視のマーカー（HTMLコメント）が付きます。このようにマークされたファイルのみ、フォルダーが変更されるたびに自動的に最新の状態に保たれます——そしてそれはOKF保管庫でのみです（ルートの`index.md`に`okf_version`があることで識別されます）。
- **読み取り専用と抜け道**——管理されているindex.mdファイルは、「このindex.mdはPlainvaによって管理され、自動的に更新されます」というバナー付きの閲覧モードで開きます。そこでは**更新**を行うか——**それでも編集する**を選択できます: これによりマーカーが削除され、そのファイルは再び完全にあなたのものになります（自動更新はされなくなります）。
- **一括処理**——**すべてのindex.mdを更新**は、保管庫のルートのコンテキストメニューと設定にあります。マーカーのないファイルはスキップされます。
- 閲覧モードでは、管理されている一覧はファイル/フォルダーアイコン付きのカードとしてレンダリングされます。リンクはPlainva内で直接開きます。

## 既存の保管庫を変換する（オプトイン）

保管庫内のファイルがOKF形式に適合していない場合（`type`フィールドの欠落、または予約名が通常のノートとして使われている場合）、Plainvaは変換を提案します——保管庫を開いたときに一度、そして**設定 → OKF → OKF変換**で恒久的に（この項目は、何かすべきことがある間のみ表示されます）。

**OKF形式に変換**ウィザードは、明確なステップで動作します。

1. **スキャン**——影響を受けるファイル数を表示します（テンプレートフォルダーとシステムフォルダーは除外されます。Frontmatterが読み取れないファイルはスキップされ、決して「修復」されません）。
2. **決定**——typeがないファイルのデフォルト`type`。既存の`type`の値は**そのまま使用**（推奨——既に有効なOKFタイプです）するか、別のフィールドに名前変更できます。
3. **プレビュー（変更なし）**——ドライランで、何が変更されるかを事前に確認できます。
4. **変換**——各ファイルは変更前に`.plainva/backups/`にバックアップされます。レポートには、何が変更され、何がスキップされたか、そしてバックアップフォルダーが要約されます。その後、任意で**index.md管理へ進む**ことができます。

ウィザードからのヒント: 変更は通常通り同期されます——Git保管庫の場合は事前にコミットしてください。

## OKFを使う必要があるか？

いいえ。OKFは緩やかな標準です。

- 新規ファイルには自動的にヘッダーが付与されます——これが邪魔になることはなく、コストもかかりません。
- 既存の保管庫（例えばObsidianからのもの）は変更なく引き続き機能します。変換は厳密にオプトインです。
- `okf_version`が欠落しているだけでは違反とは見なされません——PlainvaとObsidianを永続的に並行して使用でき、繰り返し通知されることもありません。
- Obsidianや他のあらゆるエディターは、引き続きすべてのファイルを開くことができます: それは今も昔もごく普通のMarkdownです。
